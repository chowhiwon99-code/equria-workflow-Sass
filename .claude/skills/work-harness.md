# Work Harness — 어떻게 일하는가 (구조 + 루프)

> "무엇을 만드나"는 `CLAUDE.md`/`HANDOFF.md`. "안전하게"는 `safe-changes.md`. "최신 API"는 `latest-stack.md`.
> 이 파일은 **"어떤 절차·전략으로 일하나(하네스)"** 와 **"어디까지 자동으로 검증·반복하나(루프)"** 를 정한다.

## 0. 한 줄
**조사 → 설계 → 구현 → 자동검증 → (위험 시)적대검증 → 보고.** 매 단계 사용자에게 보인다.

## 1. 작업 루프 (매 비자명 작업)
1. **조사(Explore)** — 범위가 넓거나 불확실하면 **Explore 서브에이전트**로 코드 맵을 먼저(파일·패턴·재사용 지점). 좁고 아는 파일이면 직접 읽기.
2. **설계** — 접근법 1개로 좁힘. 해법 공간이 넓으면 **여러 안 → 판정 → 합성**.
3. **구현** — `safe-changes` 원칙(추가>수정>삭제, 잘게 순차, 기존 패턴 재사용). 한 번에 큰 배치 금지.
4. **자동 검증(게이트)** — `tsc` 0 · `pnpm lint` 30/0 베이스라인(신규 0) · DDL이면 `get_advisors`+RLS 롤백 시뮬. → `/verify` 한 방. 배포 전 `next build`.
5. **적대 검증** — 위험(RLS·보안·코어 채팅/결제/멀티테넌시)하면 **N개 에이전트로 "반증해봐"** → 다수결. 그럴듯하지만 틀린 걸 거른다.
6. **보고** — 무엇·왜·검증결과(✅한 것/⬜미확인)를 분리해 사용자에게.

## 2. 멀티에이전트 사용 기준
| 상황 | 도구 |
|---|---|
| 좁은 변경·아는 파일 | 단독 |
| 넓은 조사("이름 나오는 곳 다 찾아") | **Explore 팬아웃** |
| 큰 작업(조사→설계 판정→구현→적대검증) | **Workflow**(옵트인 필요: 사용자 요청/ultracode) |
| 독립 파일 병렬 수정 | 서브에이전트 + worktree |

> 팬아웃하면 결론만 챙기고 파일덤프는 버린다. 내가 직접도 같이 돌리지 말 것(중복).

## 3. 진행상황 가시성 — 절대 원칙 (사용자 요구)
- **블랙박스 금지.** 자동화(훅·워크플로우)는 진행을 **숨기지 않는다**.
- 큰 작업은 **TaskList**로 추적(사용자가 체크리스트로 봄).
- 매 단계/Phase **보고**. 워크플로우는 `/workflows` 라이브.
- **`HANDOFF.md`(현재상태)·`WORKLOG.md`(무엇·왜·검증)가 진행 SSOT** — 작업 후 갱신.
- 자동 훅(tsc/lint) 결과도 노출.

## 4. 자동 루프 (훅 — 내가 안 봐도 검증)
- **pre-push 게이트**: `git push` 전 tsc+lint 통과 강제(실패 시 차단). 깨진 코드가 프로덕션에 못 감.
- **Stop 검증**: `src/**` 변경 턴 종료 시 `tsc` 자동 실행 → 타입에러 자동 포착.
- 훅은 **검증·게이트만**(자동수정·자동커밋 금지 = 예측가능·안정).

## 5. 명령 (반복 작업 한 방)
- **`/verify`** — tsc + lint 게이트.
- **`/deploy`** — tsc→lint→build→main 먼저 push→feat→Vercel READY 확인→HANDOFF/WORKLOG 갱신.

## 6. 배포 규칙 (요약 — 상세는 HANDOFF)
- `feat`에서 작업 → `main` push만 프로덕션. **main 먼저 push**(동시 push 시 Vercel 중복제거로 승격 누락 방지) → feat.
- DDL = `apply_migration`(MCP) + `migrations/NNN_*.sql` 파일 **둘 다**.
