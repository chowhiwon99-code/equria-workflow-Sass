# 에이전트 & MCP — 이해 · 차별화 전략 · 시한폭탄 이슈

> 상태: **생각/설계만(구현 X) · 나중에 착수.** 2026-06-10 멀티에이전트 분석(코드/라이브 검증 기반).
> 관계 문서: `PRODUCTIZATION.md`(전체 로드맵) · `HANDOFF.md`(현재 상태) · `B1-DESIGN.md`(격리).
>
> **용어/철학(중요):** 이 제품은 특정 브랜드/도메인이 아니다. 코드에 박힌 `EQURIA`·`이큐리아`·`K-뷰티`는 **판매 제품명/도메인이 아니라 "첫 번째 사내 고객(우리 회사)"의 맥락이 하드코딩된 흔적**이다(브랜드명 미정). 핵심 철학 = **"회사별 커스터마이징"**: 회사들이 AI를 업무에 쓰고 싶어도 제대로 못 붙이는 게 현실 → 우리가 **각 회사 업무에 AI가 진짜 작동하도록 맞춰주는 것**이 차별점.

## 0. 한눈 결론

- **"기본 에이전트가 멍청"한 건 모델 탓이 아니라 매뉴얼(시스템 프롬프트)이 빈약해서**다. 8개 전부 100~300단어 + 예시 0개 + 절차 없음 + 회사맥락 하드코딩. → **엔지니어링이 아니라 콘텐츠/프레임워크 문제 = 고치기 쉽다.**
- **차별점 = "회사별 커스터마이징을 잘하게 만드는 것"** (도메인 특화 아님). 우리는 이미 절반 와 있다(빌더 위저드+우수 메타프롬프트).
- **시한폭탄은 전부 "두 번째 회사" 하나에 묶여 있다:** 프리셋 자원(기본 에이전트·MCP)이 회사별 격리(B1)와 부딪힘, MCP 시크릿이 전역이라 충돌, 채팅이 MCP 워크스페이스 미검증. **두 번째 회사 받기 전 결정·해결.**
- **지금 안전하게 가치 있는 일(멀티테넌시 무관):** 기본 에이전트 매뉴얼 재작성 + 온도값 채우기 + 회사지식 수집. **리스크 0.**

## 1. 지금 에이전트/MCP가 어떻게 동작하나

**에이전트 = "특정 업무에 특화된 직원 한 명"** — 세 부분:

| 구성 | 코드 | 비유 |
|---|---|---|
| 시스템 프롬프트 | `agent_versions.system_prompt` | 업무 매뉴얼/직무기술서 (거의 전부) |
| 모델 | `agent_versions.model`(sonnet/opus) | 머리 좋기/연봉 |
| 온도·토큰 | `temperature`·`max_tokens` | 즉흥성·답변길이 한도 |

**채팅 흐름**(`api/agents/[id]/chat/route.ts`): 입력 → 현재버전(`is_current`) 매뉴얼 로드 → 최근대화+매뉴얼을 Claude에 → 스트리밍 답변 → 대화·사용량 저장.

> **핵심:** 에이전트 똑똑함의 거의 전부가 **매뉴얼(시스템 프롬프트) 품질**에서 나온다.

- **버전관리** = 매뉴얼 개정 이력(`is_current`). 좋은 설계나 기본 8개는 전부 version=1에 멈춤.
- **빌더 위저드**(`lib/agentBuilder.ts`) = 11개 구조화 질문으로 매뉴얼 자동 생성. **메타프롬프트(`SKILL_MD_SYSTEM`)는 매우 우수**(예시 중시·엣지케이스 등). 아이러니: 기본 8개는 이 빌더를 안 거치고 손으로 짧게 씀.
- **MCP** = 에이전트에 실제 도구(시트·DB·슬랙 등)를 꽂는 표준 콘센트. 관리자만 등록, 토큰은 환경변수, 연결 실패 시 조용히 스킵.

## 2. 왜 "멍청"하고 "차별성 없"는가 (8개 약점, seed.sql 진단)

세금계산서 에이전트 매뉴얼 **전문이 ~100단어**(포스트잇 수준).

1. **너무 짧다**(100~300단어 vs 고품질 1000~2000)
2. **예시(입력→출력) 0개** (빌더는 "예시가 품질 기여 최대"라 해놓고 정작 없음)
3. **단계별 절차/판단트리 없음** (업무 나열만)
4. **금지선이 추상적** (회사 고유 정책 없음 → 정책 위반 답변 가능)
5. **엣지케이스 미처리** (음수·이상치·예외 입력에서 무너짐)
6. **온도 사실상 미설정** → 전부 0.7 (정정: 컬럼·코드는 정상, seed가 값을 안 넣음 → **마이그 없이 seed 수정만**으로 해결. 단 DB 상한 1.0)
7. **성공 정의 없음** → 개선해도 측정 불가
8. **차별성 없음의 진짜 원인:** 공개정보(화장품법·부가세)만 들어가 "ChatGPT에 회사명만 박은 것". **우리 고유 지식(정책·사례·노하우·톤)이 0.**

> 결론: 둘 다 코드 갈아엎을 문제가 아니라 **프롬프트 콘텐츠 + 그걸 채우는 프레임워크** 문제 → 우리에게 유리.

## 3. 🚨 시한폭탄 (severity · 언제 · 완화) — 전부 "두 번째 회사" 트리거

| sev | 이슈 | 언제 | 완화 |
|---|---|---|---|
| 🔴 **BLOCKER** | 프리셋 자원 전략 미정 — 새 회사가 에이전트·MCP **0개 빈손**으로 시작 | 2번째 회사 온보딩 직전(B1-b) | 복제 / 공용 / 하이브리드 중 결정(§6) |
| 🔴 **HIGH→BLOCKER** | MCP 시크릿이 **전역 환경변수** → 두 회사가 같은 도구 다른 계정 연결 시 토큰 충돌/누출 | 두 회사가 동종 MCP 연결 | 시크릿을 **회사별 DB 암호화 저장**(`lib/google/crypto.ts` AES-256 패턴 재사용) |
| 🟠 HIGH | 채팅 라우트가 MCP 로딩 시 **workspace 미검증**(`.in("id",…)` only, `agent_versions.mcp_servers`가 텍스트배열이라 FK 무결성 없음) | 2번째 회사 MCP 사용 | 로딩 쿼리에 workspace 필터 + 에이전트 소속 검증 |
| 🟠 HIGH | 관리자 MCP 등록이 **항상 sentinel(우리회사)로 저장**(INSERT workspace_id 미지정) | 2번째 회사 관리자가 등록 | WorkspaceProvider로 workspace_id 명시 + 멤버 검증 |
| 🟡 MED | 버전 배포 경로·성능지표 미정 | 매뉴얼 v2 개선 시 | 기본=마이그/커스텀=빌더UI 이원화 + 피드백 루프 |

> 그 전까지는 안전. **두 번째 회사 받기 전 빨강 3개 결정·해결.**

## 4. 차별화 전략 — "회사별 커스터마이징"을 제품화

경쟁사(OpenAI Custom GPT·Claude Projects) = **"빈칸에 프롬프트 알아서 쓰세요"**. 문제: 대부분 회사가 뭘 써야 할지 모름 = 시장의 고통.

**우리 차별점:** 프롬프트를 회사가 직접 쓰게 하지 않고, **회사 맥락을 구조화 질문으로 끌어내 우리가 좋은 매뉴얼로 조립**. "AI 텍스트박스"가 아니라 **"AI를 그 회사 업무에 진짜 작동하게 하는 프레임워크"**를 판다.

**4단계 파이프라인(제품 골격):**
```
[1] 온보딩 인터뷰(회사 규모·산업·핵심업무 매핑)
 → [2] 회사 지식 캡처(매뉴얼·정책·실제 사례 저장)
 → [3] 기본 에이전트 슬롯({{회사명}}·{{정책}}·{{톤}})에 동적 주입
 → [4] 테스트&피드백(thumbs → 버전 개선)
```
**해자(moat):** 빈칸 흉내는 쉽지만 "맥락 추출 프레임워크+조립 노하우"는 누적 자산. 우리는 [3]단계(빌더+메타프롬프트)를 이미 보유, [1][2][4] 제품화만 남음. 장기엔 **에이전트 마켓플레이스**(우수 커스터마이징 공유)로 확장.

> 차별화 첫걸음 = 기본 에이전트의 `EQURIA`/`K-뷰티`를 **고정값이 아니라 슬롯**으로.

## 5. 똑똑한 + 회사별 커스터마이즈 가능한 템플릿 에이전트

발상전환: 기본 8개를 **"완성품"이 아니라 "강력한 출발점(boilerplate)"**으로 — 길고 정교하되 회사별 부분은 슬롯(`{{ }}`).

**좋은 매뉴얼 7섹션 백본:** ① 역할&경계 ② 목표&컨텍스트(슬롯) ③ 단계별 절차(판단트리) ④ 회사규칙&금지선(슬롯) ⑤ 예시 3개+ ⑥ 엣지케이스 ⑦ 출력형식&성공정의.

**예시(브랜드 무관 — CS 에이전트 백본):**
```markdown
# 역할
당신은 {{회사명}}의 고객 응대(CS) 어시스턴트입니다. "응대 초안"을 작성하며
발송 전 담당자 검토를 전제로 합니다. 환불·교환을 직접 승인하지 않고,
회사 정책을 넘는 약속을 하지 않습니다.

# 컨텍스트(회사별 슬롯)
- 회사:{{회사명}} / 업종:{{업종}} / 고객층:{{타겟고객}}
- 톤:{{톤앤매너}} / 운영정책:{{환불·교환·배송 핵심규칙}}

# 응대 절차(이 순서로 판단)
1. 문의 유형 분류 [단순질문/불만/환불·교환/배송/기타]
2. 정책({{운영정책}})에 비춰 가능/불가/조건부 판정
3. 톤({{톤앤매너}})으로 초안: 공감→사실안내→다음행동→마무리
4. 불가/모호 → 임의약속 금지, "담당자 확인 후 안내"

# 절대 금지(슬롯)
- {{금지사항}} / 개인정보 본문 노출 / 법적책임 단정 표현

# 예시
[입력] "어제 받은 상품이 파손됐어요. 환불해 주세요."
[기대출력] 사과 → {{환불정책}} 안내 → [주문번호·사진] 요청 → 담당자 안내 예고
(예시2: 배송문의 / 예시3: 정책상 불가한 환불 정중 거절)

# 엣지케이스
- 욕설/협박 → 감정자극 없이 사실만, 담당자 이관
- 정책 미명시 상황 → 임의판단 금지
- 복합문의 → 항목별 분리 답변

# 출력형식 & 성공정의
- 형식: 인사→공감→안내→다음행동→마무리(3~5문장)
- 성공: 정책위반 0, 톤일치, 고객이 다음행동을 명확히 인지
```

**온도·모델 차등(인프라 있음, seed 값만):**

| 성격 | 에이전트 | 온도 | 모델 |
|---|---|---|---|
| 정확성 | 번역·세금계산서·법무 | 0.3 | 법무=opus |
| 균형 | CS·데이터분석·문서 | 0.5~0.6 | sonnet |
| 창의 | SNS·이미지프롬프트 | 0.8~1.0 | sonnet |

**착수 순서(나중):** ① 8개를 1000단어대 재작성 ② 고정값→슬롯 ③ seed 온도 채우기 ④ 예시·엣지·성공정의 추가 ⑤ 회사지식 워크샵으로 슬롯 콘텐츠 수집(차별화 자산).

## 6. MCP 등록 + 멀티테넌시 정합 (옵션 + 권장)

문제: (1) 모든 MCP가 sentinel로 저장 (2) 시크릿 전역 충돌.

| 옵션 | 내용 | 장 | 단 |
|---|---|---|---|
| A. 완전격리 | 회사별 MCP + 회사별 암호화 시크릿 | 보안 완벽 | 공용도구도 재등록 |
| B. 전부공용 | 신뢰 MCP를 우리소유·읽기전용 노출 | 효율·중앙관리 | 회사별 시크릿 분리 불가 |
| **C. 하이브리드 ⭐** | 업계표준=공용, 사내전용=회사별 private | 장점결합·현실적 | scope 컬럼·credential 분기 구현 |

**⭐ 권장 C:** 공용층(표준 SaaS, 우리 중앙관리) + 회사층(사내 ERP/봇, 회사별 private + DB 암호화 시크릿). 로딩규칙 = `scope=PUBLIC 또는 워크스페이스 일치`. 온보딩 시 공용은 자동노출, 사내 시크릿은 별도 폼.

## 7. 나중 착수 순서 / 선행조건 (지금은 생각만)

```
[선행조건] 0. WorkspaceProvider(현재 워크스페이스 컨텍스트) — §3 빨강 전부 여기 의존

[대표 결정 — 코드보다 먼저]
1. ⭐ 프리셋 전략: 복제 / 공용 / 하이브리드  ← 가장 먼저
2. MCP 시크릿: DB 암호화 vs 환경변수(하이브리드 권장)

[병렬 가능 — 멀티테넌시 무관, 지금도 안전·리스크 0]
A. 기본 8개 매뉴얼 재작성(§5)  ← 두 번째 회사 없이도 즉시 가치
B. seed 온도값 채우기  ← 가장 쉬움(마이그 불필요)
C. 회사 지식 워크샵 → 차별화 자산 수집

[멀티테넌시 본작업 — 0번 이후]
D. MCP 저장에 workspace_id 명시 + 멤버 검증
E. 채팅 라우트 MCP 로딩에 workspace 필터
F. 프리셋 복제 RPC(1번 결정 따라)
```

**우선순위 한 줄:** (1) 지금 안전하게 가치=A·B·C. (2) 두 번째 회사 전 필수=0→1·2 결정→D·E·F. (3) 대표 지금 결정하면 좋은 것=**프리셋 전략**(권장: 초기 복제=완전격리, 장기 마켓플레이스).

### 검증한 사실 (정정 2건)
- ✅ `temperature` 컬럼 **존재**(001, default 0.7·상한 1.0)·채팅 라우트 정상 사용 → 문제는 seed 미설정뿐 → **마이그 없이 seed 수정**으로 해결.
- ✅ 채팅 라우트 MCP 로딩 **workspace 필터 부재** 확인(§3 HIGH 사실).
