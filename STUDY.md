# 바이브 코더 코스 — 우리 워크스페이스를 처음부터 끝까지

> **누구를 위한 글:** "비개발자였지만 이제 개발자 영역까지 흡수해 직접 전개하는 바이브 코더".
> **목표:** 우리가 같이 만든 *이 실제 제품*을 교재로, ① 코드가 어떻게 읽히는지 ② 우리가 *왜* 이렇게 build 했는지를 처음부터 끝까지 한 번에 이해한다.
> **읽는 법:** Part 0 → 7 순서대로 한 번 통독. 막히면 **Part 1 영어 사전**으로 돌아오면 됨. 기준일 2026-06-12.

# Part 0. 개발 세계 지도 (먼저 큰 그림)

## 0-1. "코드"란 결국 뭔가
코드는 **컴퓨터에게 시키는 지시서**를, 사람이 읽을 수 있는 영어 단어로 적은 것입니다. 그래서 코드를 읽는다는 건 결국 **영어 문장 해석**에 가깝습니다. `import`(가져와), `select`(골라), `filter`(걸러), `return`(돌려줘) — 이 단어 뜻만 알면 80%는 읽힙니다. (그 사전이 Part 1.)

## 0-2. 웹앱은 '식당' — 홀과 주방
- **프론트엔드(Frontend) = 손님이 보는 홀.** 화면·버튼·디자인. 직원이 눈으로 보고 클릭하는 모든 것.
- **백엔드(Backend) = 주방+창고+출입통제.** 데이터를 보관하고, 누가 뭘 볼지 통제하고, 계산하는 보이지 않는 부분.
- 우리가 한 일은 이 둘을 다 만든 것입니다.

## 0-3. 우리 시스템의 4대 부품 (왜 이걸 골랐나)
| 부품 | 역할 | 왜 이걸 선택했나 |
|------|------|------------------|
| **Next.js** (넥스트) | 웹앱 뼈대(홀) | 서버 그리기+브라우저 움직임을 한 틀에서. 비밀키를 손님에게 안 넘기는 구조 |
| **Supabase** (수파베이스) | 백엔드 4종(DB·로그인·파일·실시간) | 4개를 따로 사다 붙일 걸 하나로. 특히 **RLS**(보안)를 공짜로 줌 |
| **Claude + Vercel AI SDK** | AI 엔진 | 답을 한 글자씩 흘려주는 스트리밍을 표준화해 줌 |
| **Vercel** (버셀) | 배포(웹사이트로 띄우기) | 코드만 올리면 자동으로 사이트가 됨. 서버 관리 불필요 |

> 한 문장 요약: **"외부 SaaS를 매달 구독하는 대신, 같은 기능을 우리 4대 부품으로 직접 조립해 우리 것으로 가졌다."** 이게 모든 결정의 출발점입니다.

# Part 2. 코드 한 줄을 읽는 법 (실전 해석)

## 2-1. 프론트 한 조각 (알림 불러오기) — NotificationBell.tsx
```tsx
const load = useCallback(async () => {
  const { data } = await supabase
    .from("notifications").select("*")
    .eq("user_id", userId).order("created_at", { ascending: false }).limit(20)
  setItems(data ?? [])
}, [supabase, userId])
```
**한글 해석:** "`load`(불러오기)라는 함수를 만든다(기억해둔다=useCallback). 이 함수는 기다림이 있다(async). notifications**에서**(from) 전부(select \*)를, user_id가 **같은**(eq) 내 것만, created_at 최신순(order)으로 20개(limit) **가져와**(await), 비어있으면 빈 배열로(`?? []`) `items`에 **넣는다**(setItems)."

> 핵심 감각: 점(`.`)으로 이어진 건 **"그리고 또"** 로 읽으면 됩니다. "테이블 정하고(from) → 칸 고르고(select) → 조건 걸고(eq) → 정렬하고(order) → 개수 자르고(limit)".

## 2-2. 백엔드 한 조각 (권한 검사 SQL) — 059_member_role_scope.sql
```sql
create or replace function public.owner_can_set_role(target uuid)
returns boolean language sql security definer
as $$
  select exists (... 호출자가 소유한 회사에 target이 속하나 ...)
  and not exists (... target이 호출자 소유가 아닌 회사에 속하나 ...);
$$;
```
**한글 해석:** "`owner_can_set_role`(소유자가 역할을 정할 수 있나)라는 함수를 **만들거나 교체한다**(create or replace). 결과로 참/거짓(boolean)을 **돌려준다**(returns). 내용: target(대상)이 ① 내가 소유한 회사에 속하고(exists) ② 내가 소유 안 한 회사엔 **안** 속할 때(`and not exists`)만 참."
- `language sql` = 이 함수는 SQL로 쓰여짐.
- `security definer` = "이 함수는 **함수 만든 사람 권한으로** 실행" — 일반 사용자도 이 검문소를 통해서만 안전하게 일하게 하는 핵심 장치.

> 이 한 조각이 "왜 한 회사 권한을 바꾸다 남의 회사까지 새지 않나"의 정답입니다(자세한 why는 Part 4).

# Part 3. 우리가 만든 것 전부 + 왜 그렇게 했나

> 각 기능을 **무엇 → 왜 필요 → 어떻게 → 왜 이 방식(설계 결정)** 으로.

## 3-1. 전자결재 (/approval)
- **무엇:** 양식 골라 폼 작성 → 결재선(순서대로 N명) → 상신 → 순차 승인/반려 → 문서함·도장 진행.
- **왜 필요:** 기존엔 단일 admin 승인뿐 → 카카오워크식 진짜 결재선이 필요.
- **어떻게:** 문서 1줄(`approval_documents`) + 결재선 각 칸(`approval_steps`). `current_step`(지금 칸 번호)이 가리키는 사람만 도장 → 다음으로 +1.
- **왜 이 방식(결정들):**
  - *상태 변경을 서버 함수(RPC)로만* → 화면이 문서 상태를 직접 못 고치게. "내 문서를 승인완료로" 같은 조작 차단.
  - *self-approval 3중 차단* → 화면+DB규칙+서버함수 세 겹. "한 겹 뚫려도 나머지가 막는다"는 심층 방어 철학.
  - *결재선을 처음부터 N명+역할(결재/참조)로* → 지금은 수동이지만, 나중에 큰 팀의 자동 결재선도 **DB 구조 변경 없이** 확장 가능하게 미리 열어둠.

## 3-2. 회의 노트 (/meetings)
- **무엇:** 노션식 블록 에디터. `/`로 제목·표·이미지·파일 삽입, 모든 형식 인라인, 공유, AI 보조.
- **왜 필요:** 외부 노션 구독 대신 우리 안에서. 직원이 익숙한 UX.
- **어떻게:** **Tiptap**(검증된 에디터 엔진)을 조립. 본문을 HTML로 저장, 읽을 땐 다시 Tiptap에 통과(=안전 렌더).
- **왜 이 방식:**
  - *에디터를 처음부터 안 만들고 Tiptap 채택* → 안정성+개발속도. (바퀴 재발명 안 함.)
  - *저장은 HTML, 읽기는 다시 검증 렌더* → 누가 악성 스크립트를 심어도(저장형 XSS) 위험 코드를 버림. 보안과 단순함을 한 번에.
  - *AI 결과는 미리보기 후 사람이 적용* → AI 자동반영의 신뢰 문제를 차단. "사람이 최종 결정".

## 3-3. 직원 채팅 · 알림 · 실시간
- **무엇:** 1:1 채팅, 실시간 도착, 안 읽음 배지, 알림 종, 여러 창 동기화.
- **왜 필요:** 사내 소통을 슬랙 구독 없이.
- **어떻게:** Supabase **Realtime**(DB 변화→화면 즉시) + 같은 PC 여러 창은 **BroadcastChannel**(탭 간 방송) + **낙관적 업데이트**(서버 응답 전 먼저 그림).
- **왜 이 방식:**
  - *읽음 처리를 RPC(`mark_dm_read`) 하나로* → 메시지 배지와 알림을 한 번에. 예전 클라이언트 직접 처리가 RLS에 막혀 누락되던 버그를 서버 함수로 일원화.
  - *모두 읽음은 성공 시에만 배지 클리어* → 쓰기 실패 시 거짓 안심 방지(이번 리뷰 반영).

## 3-4. 공지·다중화폐·파일미리보기·사이드바
- **공지:** 대표만 작성(DB의 RLS로 강제) → 트리거가 전 직원 알림 자동 생성 → 종+배너. *왜 DB로?* 화면에서 버튼만 숨기면 우회 가능, DB에 박으면 어느 경로도 못 뚫음.
- **다중화폐:** 통화 칸 추가(기본 KRW로 기존 데이터 보존), 통화별 분리 합계. *왜 분리?* 다른 통화를 한 숫자로 합치면 환율 착시로 틀린 결과 → "있는 그대로" 보여주는 게 회계상 안전.
- **파일 미리보기:** 마우스 올리면 줌인(서명 URL로 임시 열람), 클릭하면 드로어. *왜 서명 URL?* 비공개 파일을 잠깐만 열리는 일회성 링크로 → 유출 위험↓.
- **사이드바:** 폴더 접기+방향키 이동+접힌 항목 `inert`. *왜 inert?* 안 보이는데 키보드 커서가 걸리면 접근성 깨짐 → 닫힌 영역을 '없는 셈' 처리.

# Part 4. 보안 — 우리가 가장 공들인 곳 (왜 두 번 잠그나)

## 4-1. RLS = 캐비닛 자체의 지문 인식
화면(앱) 코드와 **DB 두 군데에 이중 잠금**. 개발자가 화면에서 실수로 남의 데이터를 요청해도 DB가 마지막 관문에서 막음. **왜?** 보안을 화면 코드의 "실수 안 하기"에 맡기면 언젠가 샙니다. DB 바닥에 못 박으면 실수해도 안전.

## 4-2. 회사별 격리(멀티테넌시) — 미래 대비
모든 데이터에 `workspace_id`(회사 도장). 읽을 땐 "내 소속 회사 줄만", 쓸 땐 "내 회사 맞는지 검증". **왜 지금?** 지금은 한 회사뿐이라 체감 0이지만, 나중에 다른 회사에 팔 때 데이터가 섞이면 치명적 → 칸막이를 미리 세워둠.

## 4-3. 권한 — 오너만, 자가상승 차단
권한 변경은 **가드 트리거**가 "그 회사 오너인지" 강제 + 오너 전용 함수(`set_member_role`)로만. **왜?** 직원이 스스로 관리자가 되는 self-escalation은 가장 흔한 내부 공격.

## 4-4. 058 → 059 이야기 (왜 잠재 구멍을 미리 막았나)
- *문제:* 권한 명찰(`profiles.role`)이 회사별이 아니라 **전역**. 058 규칙은 "대상이 속한 *어떤* 회사든 내가 소유하면 변경 허용" → 멀티테넌트가 되면, 내가 소유한 B회사 멤버를 admin으로 올리면 그 사람이 내가 소유 안 한 X회사에서도 admin이 되는 누수.
- *수정(059):* "대상이 속한 **모든** 회사를 내가 소유할 때만 허용"으로 좁힘.
- **왜 지금 고쳤나(가장 중요한 교훈):** 지금은 한 회사라 **실제로 터지지 않는 잠재 결함**입니다. 그래도 고친 이유 — 보안 구멍은 "나중에 기능이 추가되는 순간" 조용히 활성화됩니다. **싸고 확실할 때 미리 막는 게 원칙.** (적대적 리뷰가 이걸 잡았고, 검증 후 배포했습니다.)

## 4-5. service_role = 마스터키는 서버에만
모든 RLS를 통과하는 키. OCR·가입 같은 관리작업에 서버에서만. **화면(브라우저)에 노출되면 누구나 전 회사 데이터를 염** → 절대 클라이언트로 안 내보냄. (그래서 AI 키도 서버 전용.)

# Part 5. 우리가 '일하는 방식'과 그 이유 (개발 철학)

> 기능보다 더 중요한 건 **어떻게 안전하게 만드느냐**입니다. 우리가 매번 지킨 원칙:

| 원칙 | 무엇 | 왜 |
|------|------|-----|
| **잘게 순차** | 한 번에 큰 배치 금지, 작업을 쪼갬 | 코드가 꼬이지 않고, 문제 생긴 지점을 바로 찾음 |
| **추가는 자유, 파괴는 검증** | 칸 추가는 안전, 삭제·변경은 신중 | 되돌릴 수 없는 변경이 사고의 90% |
| **모든 변경은 되돌리기/재현 가능** | 마이그레이션 번호, git 커밋 | 잘못돼도 1클릭 롤백·이전 상태 복원 |
| **마이그레이션으로만 DB 변경** | DB 구조는 번호 대본으로 | 추적·재현·복구. 즉흥 수정 금지 |
| **프리뷰 → 프로덕션** | 연습 무대 검증 후 실제 무대 | 한 실수가 회사 업무를 멈추지 않게 |
| **타입 strict, any 금지** | 모든 값의 모양을 못 박음 | 실수를 빌드 단계에서 잡음(런타임 사고↓) |
| **적대적 리뷰 후 배포** | 여러 시각으로 "뚫어보기" 검증 | 그럴듯하지만 틀린 코드를 거름(059가 그 예) |
| **비밀키는 서버 전용** | AI키·service_role 클라이언트 금지 | 노출되면 전 데이터 위험 |

> 이 철학이 "왜 우리가 이렇게 build 했는가"의 진짜 답입니다. **기능을 빨리 늘리면서도 라이브가 안 깨지는** 균형.

# Part 6. 전체 흐름 한 바퀴 — 직원이 버튼 하나 누르면 무슨 일이?

예: 직원이 결재 문서에서 **[승인]** 을 누름.
```
1) [홀/프론트] DocumentDetail.tsx의 onClick → supabase.rpc("act_on_approval", {문서id, "승인"})
        ↓ (인터넷)
2) [출입통제] Supabase가 "이 사람 로그인 됐나"(Auth) 확인
        ↓
3) [검문소] act_on_approval 서버함수(RPC) 실행:
     - 문서를 잠그고(동시성)  - "지금 차례가 당신 맞나"(current_step) 확인
     - 맞으면 도장 찍고 → 마지막이면 '승인완료', 아니면 다음 사람에게 알림
        ↓
4) [창고] DB의 approval_steps·documents·notifications 줄이 바뀜
        ↓ (Realtime이 변화를 감지)
5) [실시간] 연결된 모든 화면으로 "바뀌었어!" 푸시
        ↓
6) [홀/프론트] ApprovalView가 받아서 → 다시 조회 → 도장이 새로고침 없이 갱신
```
> 이 6단계가 거의 **모든 기능의 공통 골격**입니다. 화면(클릭) → 출입통제(인증) → 검문소(RPC/RLS) → 창고(DB) → 실시간 → 화면(갱신). 한 번 익히면 어느 기능이든 이 틀로 읽힙니다.

# Part 7. 바이브 코더 다음 레벨 (계속 흡수하는 법)

1. **한 파일씩 해석 연습.** 아무 컴포넌트나 열어, Part 1 사전으로 영어를 한글로 바꿔 읽기. `.`은 "그리고 또"로.
2. **"왜?"를 항상 물어라.** 코드보다 *결정*이 중요. "왜 RPC로 했지? 왜 여기서 RLS가 또 막지?" — 그 답이 실력.
3. **변경 전 3원칙:** ① 되돌릴 수 있나 ② 재현되나 ③ 프리뷰에서 먼저 확인했나.
4. **레포 문서 지도:** `HANDOFF.md`(현재 상태·배포), `WORKLOG.md`(작업 로그·왜), `B1-DESIGN.md`(격리 설계), `.claude/skills/safe-changes.md`(안전 원칙), `supabase/migrations/`(DB 대본). 여기에 실제 근거가 다 있음.
5. **질문하는 법:** "이 파일 줄별로 해석해줘 / 이건 왜 이렇게 했어? / 이거 바꾸면 뭐가 위험해?" — 이렇게 물으면 내가 같은 깊이로 풀어줍니다.
